Combat Simulator Project Constitution

NOT OFFICIAL YET!!!

Introduction

The Combat Simulator Project is a volunteer effort to create a large multiplayer network of open source combat simulators.  The project covers a broad range of subprojects, in which everyone can contribute any kind of useful resource.  This constitution provides the framework for how decisions will be made, and who can make them.  This system of project decisions will be done in a democratic way, but will also be based on merit.  There will be 5 levels of participation in the project.  Each level will have different rights and responsibilities for the direction of the project.  This system will offer greatest flexibility for managing task, and will not be dependent on any particular individual for the overall success of the project.  It is important to note that this document is only a starting point and may be further refined and improved as the project progresses.

Participation Levels

The Combat Simulator Project is highly democratic system and allows everybody to subscribe and participate on the forums.  We do however identify several participation levels, each of them of equal importance, but with different rights and responsibilities. The five levels are:

  • Passive Users - only read on the forums to learn without stepping in the way.  We invite every passive user to become an active one after feel ready for it.
  • Active Users - read and respond on the forums. Learn from others and teach others something they know (problems they had and how to work around them, bug reports, feature requests and so on). This category doesn't contribute any code but is equally important for finding bugs and to keep the project with feet on the ground. By committing code patches, active users become developers.
  • Developers - write code, patches or contribute to the project in some other ways (documentation, scripts, other resources).
  • Active Developers - are those developers who give more than one significant contribution or show enough interest in continuing the development of some part of the project and ask to become one. At this time, your request is taken into consideration and voted by the others active developers, if nobody is against it, you will be given write access to the CVS repository and you will be able to vote and rule the project.
  • Passive Developers - are those active developers who left the project or did not contribute code, votes, mails or any other resource for at least half a year. Passive developers are considered out of the project but they CVS access is never restricted unless the person asks for it or no active developer is against this in a formal voting.

Voting

Every active developer has the right to vote. Each vote can be made in one of three ways:

  • +1 = Yes, agree, or the action should be performed.  On some issues, this vote is only binding if the voter has tested the action on their own system(s).
  • 0 = Abstain, no opinion, or I am happy to let the other group members decide this issue. An abstention may have detrimental effects if too many people abstain.
  • -1 = No. On issues where consensus is required, this vote counts as a veto. All vetoes must include an explanation of why the veto is appropriate. A veto with no explanation is void. No veto can be overruled. If you disagree with the veto, you should lobby the person who cast the veto. Voters intending to veto an action item should make their opinions known to the group immediately, so that the problem can be remedied as early as possible.  Note that a member might veto the concept (with an adequate explanation), but later rescind that veto if a specific patch satisfies their objections.

An action item requiring consensus approval must receive at least 3 binding +1 votes and no vetoes. An action item requiring majority approval must receive at least 3 binding +1 votes and more +1 votes than -1 votes (i.e., a majority with a minimum quorum of three positive votes). All other action items are considered to have lazy approval until someone votes -1, after which point they are decided by either consensus or a majority vote, depending upon the type of action item.

The act of voting carries certain obligations because voting members are not only stating their opinion; they are agreeing to help on the voted issue. Since we are all volunteers, members often become inactive for periods of time in order to take care of their "real jobs" or devote more time to other projects. It is therefore unlikely that the entire group membership will vote on every issue. To account for this, all voting decisions are based on a minimum quorum.

Every active developer may request a voting on an issue by posting a message with a Subject beginning with [VOTE]

Types of Action Items

Code Freeze for Official Release

     When a vote is called to freeze the code at a particular requires a consensus.  All voters must be able to compile the code without error.  Once the code is frozen it needs to be tested.  Upon completion of successful testing, another vote must be called to approve the code for an official production release.  This also requires a consensus vote.  Upon approval of an official production release, all new features added since that last official release is now standard.

Approval for New Features

    Approval to add new features to Framework API or SISS requires a majority vote.  Anyone may add new features but they require approval before they will become part of an official release.  If the feature is disapproved it will be removed, but it may be added later if revisions satisfy voters. 

Change Request Approval

    A request to change a standard feature in Framework API and SISS requires a majority vote.  If someone would like a change in the functionality of a standard feature, a request needs to be made.  If a majority agrees that a change is necessary, a task and its priority must be given to do so.

Approval of Feature Changes

    To approve the final change of a standard feature in Framework API and SISS requires a consensus vote.  Due to that face that changes to standard features may impact many programs and functions, rigorous testing will be required before such changes will be approved.

Change Task Priorities

    To change the priority of any CSP task requires a majority vote.

Change of Project Goals

    To make a change to CSP goals requires a majority vote.

Constitution Changes

     To make any amendments to this constitution requires a consensus vote.

    Patch Approval

     To approve official bug patches requires a majority vote.

Committing

Each developer is responsible for notifying everyone via the forum when they have an idea for a new feature or major change to propose for the product. The distributed nature of the Combat Simulator Project requires an advance notice of 72 hours in order to properly review a major change 

Related changes should be committed as a group, or very closely together. Half-completed projects should not be committed unless doing so is necessary to pass the baton to another developer who has agreed to complete the project in short order. All code changes must be successfully compiled on the developer's platform before being committed.

The current source code tree should be capable of complete compilation at all times. However, it is sometimes impossible for a developer on one platform to avoid breaking some other platform when a change is committed, particularly when completing the change requires access to a special development tool on that other platform. If it is anticipated that a given change will break some other platform, the committer must indicate that in the commit log.

The committer is responsible for the quality of any third-party code or documentation they commit to the repository. All software committed to the repository must be covered by the LICENSE specific for that project or contain a copyright and license that allows redistribution under the same conditions.

A committed change must be reversed if it is vetoed by one of the voting members and the veto conditions cannot be immediately satisfied by the equivalent of a "bug fix" commit. The veto must be rescinded before the change can be included in any public release.

Patches

When a specific change to the software is proposed for discussion or voting on the mail list, it should be presented in the form of input to the patch command. When sent to a mail list, the message should contain a Subject beginning with [PATCH] and a distinctive one-line summary corresponding to the action item for that patch along with the names of the contributors.
